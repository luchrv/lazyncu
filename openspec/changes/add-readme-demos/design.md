# Design: add-readme-demos

## Context

lazyncu's README is complete but text-only. lazygit's README (the reference) embeds one compressed GIF per feature, hosted in-repo, generated by scripts rather than recorded by hand. lazyncu is a tview TUI that shells out to `ncu` and `npm audit`; a demo needs real scannable projects and network access. The repo is public; releases exist since v0.1.0.

Explore session decisions (2026-07-23): keep current README structure (no full lazygit replica), 3 GIFs (hero, vulns, add-path), VHS scriptable capture with a written guide, typical Go badges + Homebrew, synthetic fixtures.

## Goals / Non-Goals

**Goals:**
- A visitor sees what lazyncu does within seconds of opening the README.
- Every GIF is regenerable with one command; captures never depend on personal machines' configs or paths.
- A contributor can add a new demo following `docs/DEMOS.md` unassisted.

**Non-Goals:**
- Full lazygit-style README restructure (TOC, per-feature sections, FAQ, Alternatives).
- CI-automated demo regeneration (VHS in CI needs fonts/ttyd setup — not worth it for 3 GIFs; manual `make demos` suffices).
- Videos/MP4 or asciinema hosting — GIFs in-repo like lazygit.

## Decisions

### D1 — VHS over manual recording or asciinema

charmbracelet VHS: declarative `.tape` files (`Type`, `Sleep`, `Set FontSize`, …) rendered headlessly to GIF with consistent theme, font, and dimensions. Tapes are code — versioned, reviewable, regenerable.

- *Alternative — manual recording (Kap, t-rec)*: fast once, irreproducible; every UI tweak means re-recording by hand. Rejected.
- *Alternative — asciinema + agg*: records real sessions, replay-accurate, but capture is still interactive typing; agg theming is more limited. Rejected.

### D2 — Three tapes, one GIF each

| Tape | Shows | README placement |
|---|---|---|
| `hero.tape` | launch, parallel scan streaming in, selecting source/project | right after the intro bullet list |
| `vulns.tape` | `v` toggle, severity counters, dependency chain detail | "Audit coverage notes" section |
| `add-path.tape` | `a`, typing a path, immediate scan of new source | "Configuration" section |

Output `assets/demo/<name>.gif`. Tapes live in `assets/tapes/`. GIF budget: target < 3 MB each (VHS `Set` width/height ~1200×700, moderate framerate, demos ≤ ~25 s).

### D3 — Config isolation via `XDG_CONFIG_HOME`

lazyncu reads `$XDG_CONFIG_HOME/lazyncu/config.toml`. Each recording session prepares a throwaway config dir (seeded `config.toml` registering the fixture paths) and the tape exports `XDG_CONFIG_HOME` before launching `./lazyncu`. Recording never touches `~/.config/lazyncu` and GIFs never leak personal paths. A small setup script (`assets/tapes/setup-demo-env.sh`, invoked by `make demos`) builds the binary, creates the temp config, and installs fixture `node_modules`/lockfiles as needed.

### D4 — Synthetic fixtures under `demo/fixtures/`

Two projects, enough to show a tree without bloating scan time:

- `demo/fixtures/webapp/`: npm project, `package.json` pinning outdated deps across severities (major/minor/patch mix — e.g. old `express`, `chalk`, `commander`) plus a known-vulnerable pin (`lodash@4.17.15`) so `npm audit` yields critical/high rows and a dependency chain.
- `demo/fixtures/tools/`: second, smaller npm project so the tree shows multiple sources scanning in parallel.

Lockfiles committed (audit requires them). `node_modules` NOT committed — setup script runs `npm install` in fixtures before recording. Fixture `package.json`s marked `"private": true`.

- *Alternative — record against real user projects*: leaks paths, unreproducible. Rejected in explore.

### D5 — Badges: shields.io row at README top

GitHub release (latest tag), Go version (`go-mod`), License Apache-2.0, GitHub downloads (all releases), static Homebrew tap badge linking the tap repo. Implementation finding: goreportcard.com is retired (badge renders "go report: retired") — replaced by the Go version badge. `shields.io/homebrew/v` only covers homebrew-core, hence the static tap badge.

### D6 — `make demos` + `docs/DEMOS.md`

Makefile target: `demos` → runs setup script, then `vhs` on each tape. Guide covers: install (`brew install vhs`), how a tape is structured (annotated hero.tape walkthrough), regenerating all demos, adding a new demo (copy tape, adjust keys/timing, reference GIF in README), GIF size guidance, and the network requirement.

## Risks / Trade-offs

- [GIF content ages (more pending updates over time; audit output shifts)] → accepted; regenerate with `make demos` when it looks stale. Fixtures pin exact versions so the *structure* of the demo is stable even if counts drift.
- [Recording needs network; registry hiccups make scans slow/empty] → tapes use generous `Sleep` margins after launch; guide notes to re-run on flaky output.
- [GIF binaries bloat git history on every regeneration] → only 3 files, target < 3 MB each, regenerate rarely (visual changes only). If it ever hurts, move to a git LFS or release-asset scheme — not now (YAGNI).
- [`lodash@4.17.15` advisories could be withdrawn or npm audit format shift] → any old package with published advisories works; guide tells how to pick a replacement (`npm audit` locally on the fixture).
- [VHS rendering varies by font availability across machines] → tape pins `Set FontFamily`/`Set FontSize`/`Set Theme`; guide lists them as prerequisites.
